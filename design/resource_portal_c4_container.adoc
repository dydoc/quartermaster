== Overview

The Resource Portal is deployed as a set of Kubernetes workloads on a Rancher managed cluster. At this level the system decomposes into two runtime processes — the **Web Application** and the **Approval Controller** — and three **Custom Resource Definitions** that live on the management cluster (the cluster where the Resource Portal runs) and act as the shared, durable data layer between them.

The boundary is the management cluster itself. There are no databases, message queues, or external storage services: Kubernetes etcd (via the CRD API) is the single source of truth for all portal state.

A key security boundary runs through this diagram: **user identity is used only for operations on the management cluster** (Web App authenticating as the logged-in user via Rancher), while **the Approval Controller acts under its own dedicated service account** when applying CRDs on target downstream clusters. End users hold no permissions on those clusters for the resource types exposed by the portal. This is the architectural guarantee that prevents bypassing the approval workflow.

== Containers

=== Web Application

The Web Application is a monolith that serves all HTTP traffic. It is server-side rendered — there is no separate frontend bundle or SPA framework. A single process handles session management, OIDC authentication, catalogue browsing, request submission, approval actions, and status display.

At startup the Web App launches a **informer** for both `ResourceRequest` and `ProvisionedResource` objects. The informer maintains a local in-memory cache kept in sync by the Kubernetes watch mechanism. When a user navigates to their resource list, the Web App reads directly from that cache — no polling, no per-request API calls to the cluster.

At submission time the Web App is responsible for **resolving the approval data**: it reads the relevant `ResourceCatalogue` entry, evaluates the multi-stage approval policy (which may reference IdP groups, team labels, or explicit user lists), and populates per-stage group identifiers and user identities via the Approver Resolver. The Approval Controller consumes this data as structured state — it never re-evaluates routing rules or queries any identity service itself.

=== Approval Controller

The Approval Controller is a standard Kubernetes controller built with `controller-runtime`. It watches `ResourceRequest` objects and reacts to status transitions driven by Approver actions in the Web App.

The controller's reconcile loop is deliberately policy-agnostic: the approval policy is carried as structured data inside the `ResourceRequest` spec and evaluated at submission time by the Web App. The controller reads the pre-resolved approver list and the current approval decisions, and advances the state machine accordingly — without any hardcoded routing logic.

The controller's **only write to a target cluster** is creating, updating, or deleting the **CRD target instance** (e.g. a `NamespaceRequest`), authenticated as its own dedicated Kubernetes service account through Rancher. It never touches the derived objects that the CRD target instance produces — Pods, Services, ConfigMaps, or any other resource. Those are the exclusive responsibility of the CRD target controller on the target cluster.

In addition to driving provisioning, the controller runs a **Health Observer** on every reconcile pass for all `ProvisionedResource` objects in `provisioned` or `updating` phase. The Health Observer reads the `status.conditions` of the CRD target instance on the downstream cluster (read-only via Rancher) and updates `ProvisionedResource.status.phase` to `degraded` or `unknown` if the instance has become unhealthy or has disappeared. If the CRD target instance was deleted out-of-band — not through a portal `delete` request — the Health Observer re-applies it via SSA to restore it. This is the only form of drift correction the Approval Controller performs, and it operates at the CRD target instance level only, never on derived objects.

NOTE: The Health Observer re-applies a deleted CRD target instance directly, without creating a `ResourceRequest`. This is a deliberate trade-off: the recovery is fast and automatic, but it bypasses the approval workflow and produces no audit trail entry beyond the controller's own logs. Operators who require an approval trail for all cluster writes — including automated recovery — should disable the re-apply behaviour and instead alert on `ProvisionedResource.status.phase = unknown`, treating out-of-band deletion as a manual incident.

=== Integrity of derived objects: SSA with field ownership and drift reconciliation

Applying the CRD target instance is only the first step. The CRD target controller on the downstream cluster will in turn create derived objects — Pods, Services, ConfigMaps, PVCs, or any other Kubernetes resource the resource type requires. An end user with standard namespace permissions could modify those derived objects directly, bypassing the portal's governance entirely.

The mechanism that prevents this is **not** something the Approval Controller does. It is a property of the CRD target controller — and it is enforced through the onboarding contract (defined in the <<_contract_for_crd_target_authors>> section below), not at runtime by the portal.

**Server-Side Apply (SSA) with field ownership.** The CRD target controller applies all derived objects using SSA, declaring itself as the field manager on every configuration-significant field. Once a field is owned by the CRD target controller's field manager, any attempt by another actor to patch that field without using `--force` is rejected by the API server with a conflict error. This makes out-of-band modification of critical fields fail immediately and visibly.

**Active drift reconciliation.** The CRD target controller does not treat the initial apply as a one-time operation. On every reconcile loop — triggered by watch events or periodic resync on the target cluster — it re-applies the full desired state of all derived objects via SSA. If a user has forced a field override, the next reconcile restores the correct state. The reconcile window is the only gap in protection; under normal conditions it is measured in seconds.



**The accepted trade-off.** This strategy accepts a reconcile window during which a forced modification to a derived object is live before being corrected by the CRD target controller. The strategy does not cover `DELETE` of derived objects synchronously — if a user deletes a Pod it will be recreated on the next reconcile pass of the CRD target controller, but it is absent during that window.

=== Contract for CRD target authors

Because the integrity guarantees depend on how each CRD target controller is implemented, the portal defines an onboarding contract that every CRD type must satisfy to be admitted to the catalogue:

* all derived objects must be created and updated via SSA, never with imperative `Create` followed by silent drift
* field ownership must be declared on every configuration-significant field, not just top-level spec fields
* the reconcile loop must compare observed vs desired state on all owned fields, not only check object existence
* CRD target instance names on downstream clusters must be derived from the UID of the originating `create` `ResourceRequest` (not from user-provided names), and must be kept under 63 characters after any prefix — Kubernetes name length limits apply and many admission webhooks impose stricter bounds

=== ResourceRequest CRD

`ResourceRequest` is the central object in the system. It represents the full lifecycle of a single resource request and carries all state needed by both the Web App and the Approval Controller without any further lookup.

The `action` field distinguishes three semantics: `create` (first provisioning), `update` (modification of an already-provisioned resource), and `delete` (deprovisioning). For `update` and `delete`, the `targetRef` field identifies the `ProvisionedResource` being acted upon, carrying both name and UID — the UID prevents the controller from acting on a different object that happens to share the same name.

For `update` actions, the Web App snapshots the current `appliedParameters` from the `ProvisionedResource` into a `previousParameters` field at submission time. This lets the approver see exactly what is changing and gives the controller a deterministic rollback target if the update fails. The snapshot also includes the `generation` of the `ProvisionedResource` at the time of the snapshot — the controller validates this at reconcile time to detect whether another spec update completed between submission and approval, making the rollback target stale. `generation` is used rather than `resourceVersion` so that status-only writes (e.g. the Health Observer updating `status.phase`) do not invalidate the rollback target. This requires the `ProvisionedResource` CRD to have the status subresource enabled, which separates spec writes (increment `generation`) from status writes (do not).

Fields carried by a `ResourceRequest`:

* `spec.action` — `create` | `update` | `delete`
* `spec.targetRef.name` / `spec.targetRef.uid` — present for `update` and `delete`
* `spec.resourceType` — references a `ResourceCatalogue` entry by name
* `spec.targetCluster` — name of the target cluster resolved from `ResourceCatalogue.spec.clusterSelector` at submission time; the controller addresses the downstream cluster directly from this field without reading the catalogue
* `spec.targetCRD` — snapshot of `{group, version, kind, instanceNamePrefix}` from `ResourceCatalogue` at submission time; the controller uses the GVK to construct and apply the CRD target instance
* `spec.parameters` — the desired configuration
* `spec.previousParameters` — snapshot of current configuration at submission time (update only)
* `spec.previousGeneration` — `generation` of the `ProvisionedResource` at snapshot time; used by the controller to detect stale rollback targets (update only). `generation` increments only on spec writes, not status patches, so Health Observer updates do not cause false positives
* `spec.approvalPolicy` — full snapshot of `ResourceCatalogue.spec.approvalPolicy` at submission time, including all stage definitions, `dependsOn` relationships, and `groupsClaim`; the controller evaluates the DAG from this snapshot without reading the catalogue
* `spec.approvalStages[]` — per-stage resolved approval data, populated at submission time by the Approver Resolver:
** `.name` — stage name from the policy
** `.resolvedGroups[]` — OIDC group identifiers for this stage: copied directly from `idpGroup` entries; resolved from the target cluster's label (and optionally transformed via `groupClaimTemplate`) for `teamLabel` entries; checked against the logged-in user's `groupsClaim` at approval time
** `.resolvedUsers[]` — individual user identity strings for this stage, sourced from `explicitUsers` entries; checked against the logged-in user's OIDC `email` or `sub` at approval time
** `.requiredApprovals` — minimum number of distinct user approvals to complete this stage
* `spec.requester` — identity of the user who submitted the request, extracted from the OIDC token at submission time; used by the Notifier to address state-change emails; immutable after creation
* `status.phase` — `pending` → `approved` / `rejected` → `provisioning` → `provisioned` / `rolling_back` → `failed` / `rollback_failed`
* `status.notificationSent` — boolean set by the Notifier after a notification is successfully delivered; prevents duplicate notifications on controller restart
* `status.approvalStages[]` — per-stage approval state:
** `.name` — stage name
** `.state` — `pending` | `active` | `completed` | `rejected`
** `.decisions[]` — one entry per approve/reject action: `{approver, timestamp, action: approve|reject}`
* `status.failureReason` — populated on `failed` and `rollback_failed`
+
NOTE: `provisioned` is the terminal success phase for all three action types. For `create` and `update` it means the resource is operational on the target cluster. For `delete` it means the deprovisioning action completed successfully — the resource no longer exists. The same phase name is used deliberately: it signals that the _request_ reached its intended terminal state, regardless of what that state implies for the resource's existence.
+
NOTE: `rolling_back` is an `update`-only intermediate phase, visible to the End User in the portal. The controller is re-applying `previousParameters` to restore the CRD target instance. `ProvisionedResource` retains `appliedParameters` throughout.
+
NOTE: `rollback_failed` is a terminal phase reached when rollback retries are also exhausted. The cluster state is unknown — the CRD target instance may be in a partially updated state. Manual intervention is required. The Health Observer will detect and surface `degraded` or `unknown` status on `ProvisionedResource` on the next reconcile pass.



=== ProvisionedResource CRD

`ProvisionedResource` is the management-cluster's authoritative record of a resource that has been successfully provisioned on a target cluster. It is written and owned exclusively by the Approval Controller — the Web App and users never write to it directly.

The controller updates `ProvisionedResource` only after observing that the CRD target instance on the downstream cluster has reached a `Ready` state — not optimistically at the moment of apply. This means `ProvisionedResource` always reflects the last confirmed operational state. During an in-flight update the object retains the previous `appliedParameters` until the new ones are confirmed; if the update fails the controller uses those parameters as the rollback target.

The `ProvisionedResource` carries a finalizer (`portal.example.com/deprovisioning`) added by the controller at creation time. The finalizer is removed only after the controller has confirmed that the CRD target instance on the downstream cluster has been successfully deleted. On each reconcile invocation in the delete path, the Provisioning Handler reads the CRD target instance — if it receives `NotFound` (e.g. after a crash-restart or on first check), the handler treats this as confirmed deletion and proceeds directly to finalizer removal, making the delete path fully idempotent.

Fields carried by `ProvisionedResource`:

* `spec.resourceType` — the CRD type that was provisioned
* `spec.targetCluster` — the downstream cluster where the resource lives
* `spec.targetRef` — GVR + namespace + name of the CRD target instance
* `spec.appliedParameters` — the parameters currently confirmed operational
* `status.phase` — `provisioned` | `updating` | `degraded` | `unknown` | `deleting`
* `metadata.labels`:
** `portal.example.com/resource-type` — for filtering by type in the informer cache
** `portal.example.com/created-by-request` — UID of the originating `create` ResourceRequest
** `portal.example.com/last-updated-by-request` — UID of the most recent successful `update` ResourceRequest

The label-based cross-reference allows the Web App to retrieve the full history of `ResourceRequest` objects associated with a given `ProvisionedResource` via a single label selector query on the informer cache, without any additional API server calls.

The cross-reference is bidirectional. `ResourceRequest` objects for `update` and `delete` actions carry a complementary label:

* `portal.example.com/target-provisioned-resource` — UID of the `ProvisionedResource` being acted upon

This allows the Approval Controller to list all `ResourceRequest` objects that reference a given `ProvisionedResource` — used by the Concurrency Guard — without any additional API server calls.

=== ResourceCatalogue CRD

`ResourceCatalogue` defines what resources are available for request. It is cluster-scoped, has no `status` subresource, and has no controller loop — it is pure admin-written configuration. The Web App reads it on demand via direct API calls (not the informer cache) because it is low-frequency, admin-written, and consulted only at submission time. The Approval Controller never reads it.

==== Fields

`spec.displayName`::
Human-readable name shown in the catalogue browsing UI. Required.

`spec.description`::
Optional longer description shown to End Users at submission time.

`spec.targetCRD`::
The CRD type to provision on target clusters. All sub-fields are required.
+
* `group`, `version`, `kind` — GVK of the CRD target instance on the downstream cluster
* `instanceNamePrefix` — prefix prepended to the UID of the originating `create` `ResourceRequest` to form the target instance name. CEL constraint: `size(instanceNamePrefix) + 36 ≤ 63` (Kubernetes DNS label limit). The prefix should include the separator character (e.g. `"ns-"`, not `"ns"`).
+
Snapshotted into `ResourceRequest.spec.targetCRD` at submission time.

`spec.clusterSelector`::
Standard Kubernetes `LabelSelector`. Used exclusively at submission time by the Submission Handler to find eligible Rancher cluster objects.
+
* Zero matches → submission fails with a user-facing error
* One match → target cluster auto-selected
* Multiple matches → user selects from the submission form
+
The selected cluster name is written to `ResourceRequest.spec.targetCluster`. `clusterSelector` is never snapshotted. The controller uses the concrete cluster name in the `ResourceRequest` directly.

`spec.parameterSchema`::
An embedded OpenAPI v3 schema defining the parameters an End User may submit. Cross-field validation uses `x-kubernetes-validations` CEL rules within the schema. The field uses `x-kubernetes-preserve-unknown-fields: true` in the CRD definition because the schema structure cannot be predicted at CRD definition time.
+
Snapshotted in full into `ResourceRequest.spec.parameterSchema` at submission time for audit.

`spec.quota`::
Optional. Both `perUser` and `perTeam` are independently optional.
+
* `perUser.maxActive` — max `ResourceRequest` objects in non-terminal phase per user per resource type
* `perUser.maxProvisioned` — max `ProvisionedResource` objects per user per resource type
* `perTeam.teamClaim` — required when `perTeam` is present; OIDC claim name that identifies the user's team; must resolve to a single string value (multi-valued claim → submission fails with a clear error)
* `perTeam.maxActive` / `perTeam.maxProvisioned` — same limits at team level
+
Quota checks are performed by the Submission Handler reading the informer cache at submission time (soft enforcement — see Key Design Decisions). Not snapshotted.

`spec.approvalPolicy`::
The multi-stage approval policy governing who must approve requests of this type. Snapshotted in full into `ResourceRequest.spec.approvalPolicy` at submission time.
+
* `groupsClaim` — required when any stage uses a group-type approver. The OIDC claim name whose values are checked for group membership at approval time (e.g. `"groups"`, `"roles"`). Configurable per catalogue entry because claim names vary across IdP configurations.
* `stages[]` — ordered list of approval stages forming a DAG. A stage with an empty `dependsOn` list becomes active immediately on submission. A stage becomes active only when all stages in its `dependsOn` list reach `completed` state. All stages must complete for the overall request to be approved.
+
Each stage:
+
** `name` — unique within the catalogue; pattern `[a-z][a-z0-9-]*`; appears in `status.approvalStages` and is visible to users in the portal
** `dependsOn[]` — names of stages that must complete before this stage activates. A stage may only reference stages that appear earlier in the YAML list — the structural ordering constraint that prevents DAG cycles without graph traversal
** `requiredApprovals` — minimum number of distinct user approvals (N-of-M) to complete this stage; minimum 1
** `approvers[]` — one or more approver entries merged into a single pool per stage. Three types:
+
`type: idpGroup` with `values: []string` (required, min 1);;
OIDC group identifiers. Membership is checked against the logged-in user's `groupsClaim` at the moment of the approval action. No group expansion is performed at submission.
+
`type: explicitUsers` with `values: []string` (required, min 1);;
Individual user identity strings in the same format as the OIDC `email` or `sub` claim used throughout the system.
+
`type: teamLabel`;;
Reads the label at `labelKey` (optional; default `"portal.example.com/team"`) from the Rancher cluster object for the selected target cluster and records the value as a group identifier for this stage.
Optional `groupClaimTemplate`: a template string using `{team}` as a placeholder, applied to the label value to produce the final group identifier (e.g. `"cn={team},ou=groups,dc=example,dc=com"` for environments where the OIDC `groupsClaim` carries full DNs but cluster labels carry only the team name).
Optional `values`: allowlist of valid label values; if non-empty, a label value absent from the list causes this approver entry to contribute zero identities to the stage pool.

==== Admission webhook

A `ValidatingAdmissionWebhook` enforces constraints on `ResourceCatalogue` objects that cannot be expressed in CEL schema validation alone:

* Stage names are unique within the `stages` list
* Every `dependsOn` entry names a stage that exists
* Every `dependsOn` entry names a stage that appears earlier in the list (structural ordering constraint)

Schema-level constraints (`instanceNamePrefix` length, `requiredApprovals ≥ 1`, `name` pattern, `teamClaim` required when `perTeam` is present) are enforced by CEL rules in the CRD definition itself.

=== Mail Relay

An SMTP relay — either an in-cluster relay or an external SMTP gateway — used exclusively by the Approval Controller to deliver state-change notifications. The relay has no knowledge of portal state; it receives a plain SMTP message and delivers it. Retry and backoff are handled by the controller, not the relay.

== Data Flows

=== Happy Path — create

1. **End User** logs in via OIDC (Web App ↔ Identity Provider).
2. **End User** browses the catalogue (Web App reads `ResourceCatalogue`).
3. **End User** submits a request. Web App resolves the approver list, creates a `ResourceRequest` (`action: create`) in `pending` phase.
4. **Approver** sees the pending request, approves it. Web App records the decision in the `ResourceRequest`.
5. Approval Controller detects the transition, advances phase to `provisioning`, applies the target-cluster CRD via Rancher as controller service account, and exits with `RequeueAfter`.
6. On each subsequent reconcile re-invocation, the controller reads the CRD target instance `status.conditions`. Once `Ready` is confirmed, it writes `ProvisionedResource` with `appliedParameters`, advances `ResourceRequest` phase to `provisioned`, and sends a notification email.
7. **End User** sees `provisioned` status from the informer cache, no polling.

=== Happy Path — update

1. **End User** selects an existing provisioned resource and submits a modification.
2. Web App checks that no other `ResourceRequest` is in a non-terminal phase for the same `ProvisionedResource`. If one exists, submission is rejected with an explicit message.
3. Web App snapshots `appliedParameters` and `generation` into `previousParameters` and `previousGeneration`, creates a `ResourceRequest` (`action: update`, `targetRef` with UID).
4. **Approver** sees a diff view (new parameters vs `previousParameters`), approves it.
5. Controller validates `previousGeneration` against the current `ProvisionedResource.metadata.generation`. If they differ, another spec update completed between submission and approval — the controller sets phase to `failed` with `failureReason = stale rollback target` rather than proceeding with a potentially incorrect rollback target.
6. Controller advances to `provisioning`, applies the updated CRD target instance via SSA, and exits with `RequeueAfter`.
7. On each subsequent reconcile re-invocation, the controller reads the CRD target instance `status.conditions`. On confirmed `Ready`, it updates `ProvisionedResource` with new `appliedParameters` and sends a notification.
8. **End User** sees updated state from informer cache.

=== Failure Path — update with rollback

1. Controller applies updated CRD target instance; downstream controller fails to reconcile.
2. Controller retries with exponential backoff.
3. After exhausting retries, controller sets `ResourceRequest` phase to `rolling_back` — visible to the End User in the portal — and begins re-applying `previousParameters` to restore the CRD target instance to its pre-update state via SSA.
4. `ProvisionedResource` is never modified — it still carries the last confirmed `appliedParameters`.
5. Once the rollback apply is confirmed `Ready` on the target cluster, controller sets `ResourceRequest` phase to `failed` with `failureReason`, sends notification email.
6. Web App shows `failed` on the update request; `ProvisionedResource` still shows `provisioned` with the previous parameters — no inconsistent state visible to the user.

=== Failure Path — rollback also fails

1. Rollback retries are exhausted: the CRD target instance cannot be restored to `Ready`.
2. Controller sets `ResourceRequest` phase to `rollback_failed` with `failureReason`.
3. `ProvisionedResource` is not modified — it retains the last confirmed `appliedParameters`.
4. On the next reconcile pass, the Health Observer reads the CRD target instance `status.conditions` and sets `ProvisionedResource.status.phase` to `degraded` or `unknown`, making the situation visible in the portal.
5. Manual intervention is required to restore the cluster to a consistent state.

=== Deprovisioning

1. **End User** requests deletion of a provisioned resource.
2. Web App creates a `ResourceRequest` (`action: delete`, `targetRef` with UID).
3. After approval, controller advances to `provisioning`, sets `ProvisionedResource` phase to `deleting`.
4. Controller checks whether the CRD target instance exists on the downstream cluster. If it is already absent (e.g. deleted out-of-band, or the controller crashed after a previous deletion), it proceeds directly to finalizer removal — the delete path is fully idempotent.
5. If present, controller deletes the CRD target instance via Rancher and exits with `RequeueAfter`. On each subsequent reconcile re-invocation the controller reads the object; once absent (`NotFound`), it proceeds.
6. On confirmed absence, controller removes the finalizer from `ProvisionedResource` — the GC deletes it.
7. Controller advances `ResourceRequest` phase to `provisioned` (the delete action completed successfully), sends notification.

== Key Design Decisions

**Dual identity, explicit boundary.** The Web App uses the logged-in user's identity for all _write_ operations on the management cluster (creating `ResourceRequest` objects, recording approver decisions) so that every user action is fully auditable through standard Kubernetes audit logs. _Read_ operations — the informer cache watch connection — use the Web App's own service account; a `SharedInformer` is a single long-lived connection and cannot carry per-user identity. Object-level access filtering for the informer-served data is enforced at the application layer within each handler (see "Informer cache access control" below). The Approval Controller uses a dedicated service account with narrow RBAC on target clusters for the provisioning step. End users have no `create`/`patch` permissions on the CRD types managed by the portal on those clusters. The portal is not a convenience layer over existing user permissions — it is the only path.

**SSA field ownership + drift reconciliation over admission webhook.** The integrity of derived objects on target clusters is enforced through the CRD target controller using Server-Side Apply with explicit field ownership combined with active drift reconciliation on every resync — not by the Approval Controller and not by a ValidatingAdmissionWebhook. The Approval Controller's scope ends at the CRD target instance; everything below that is the CRD target controller's responsibility, enforced by the onboarding contract. The trade-offs accepted: the reconcile window (seconds) is the gap in protection, and synchronous DELETE coverage is not provided. In exchange, there is no additional webhook component to operate on each target cluster, no Fail Open/Fail Closed availability dilemma, and no bootstrap ordering dependency.

**CRD target onboarding contract.** Because the integrity guarantees depend on correct implementation of SSA and drift reconciliation in each CRD target controller, the portal defines an explicit onboarding contract. This contract is the boundary between what the portal can guarantee and what it delegates to the CRD target author.

**`rollback_failed` as a distinct terminal phase.** When rollback retries are exhausted the cluster state is genuinely unknown — the CRD target instance may be partially updated. Using `failed` for both "rollback confirmed" and "rollback failed" would hide this distinction from the operator. `rollback_failed` is a signal that manual intervention is required, not merely a retry.

**CEL immutability on governance fields.** `spec.approvalStages`, `spec.approvalPolicy`, `spec.targetCluster`, `spec.targetCRD`, `spec.requester`, `spec.action`, and `spec.targetRef` on `ResourceRequest` are immutable after creation. This prevents any actor — including the Web App itself — from retroactively modifying the governance perimeter of an in-flight request.

**`status.notificationSent` prevents duplicate notifications.** The Notifier sets this boolean after successful delivery. On controller restart, the Reconciler checks this field before invoking the Notifier, ensuring at-most-once delivery without external state. A notification that was in-flight at restart time may be lost (fire-and-forget semantics are preserved), but a completed transition is never re-notified.

**`previousGeneration` guards against stale rollback targets.** The Web App snapshots `ProvisionedResource.metadata.generation` at submission time. If another spec update completes and is confirmed between submission and approval, the controller detects the mismatch and fails the request rather than proceeding with a rollback target that no longer reflects the actual pre-update state. `generation` is preferred over `resourceVersion` because the Health Observer writes status-only patches (updating `status.phase` without touching spec), and those patches increment `resourceVersion` but not `generation` — using `resourceVersion` would cause false-positive stale-target failures on every Health Observer pass during the approval window.

**Deprovisioning is idempotent on `NotFound`.** On each reconcile invocation in the delete path, the Provisioning Handler reads the CRD target instance. If it receives `NotFound`, it treats this as confirmed deletion and proceeds directly to finalizer removal — no special-case logic needed for crash-restart recovery.

**Monolith, not microservices.** The Web Application is intentionally a single deployable binary. The operational surface is kept small: one Deployment, one Service, no inter-service networking inside the portal boundary. The separation of concerns is internal (packages, not processes).

**Policy as data, not code.** The approval policy lives in `ResourceCatalogue` and is snapshotted into `ResourceRequest` at submission time. The Approval Controller is a pure state machine that never contains routing logic. Routing rules can be changed without redeploying the controller, and historical requests always reflect the policy that was in effect when they were submitted.

**Per-stage approval data resolved at submission; group membership checked live at approval.** The Approver Resolver is called once per submission and populates `spec.approvalStages` with OIDC group identifiers and individual user identities — no group expansion to a flat user list is performed. For `idpGroup` and `teamLabel` approvers, group membership is checked live against the logged-in user's current OIDC token at the moment they submit an approval decision. This means a user who loses group membership after submission immediately loses their approval right on pending requests — revoked access takes effect without waiting for a new submission. The Approval Controller never queries any identity service; it evaluates only stage state, decision counts, and the DAG structure from `spec.approvalPolicy`.

**Informer, not polling.** The Web App uses a informer to maintain a live cache of `ResourceRequest` and `ProvisionedResource` objects. Status updates appear in the UI within the informer's resync period, driven by Kubernetes watch events, with no per-request API calls.

**Three CRDs, all on the management cluster.** There is no portal-specific database. The management cluster's etcd is the authoritative store for all portal state. `ResourceRequest` and `ProvisionedResource` are controller-owned; `ResourceCatalogue` is admin-owned.

**Failure is a first-class state.** `failed` and `rollback_failed` are terminal phases with a structured `failureReason`. The controller retries with backoff before reaching either. Once in a terminal phase, the request is immutable — a new request must be submitted. This avoids ambiguous partial-apply states.

**Informer cache access control is application-layer, not RBAC-layer.** The Web App's `SharedInformer` uses the Web App's own service account for its single watch connection — it sees all `ResourceRequest` and `ProvisionedResource` objects. Per-user visibility is enforced in application code: the Status Handler filters objects by the requester identity stored in `spec.requester`; the Approver Handler filters by whether the logged-in user is eligible for any active approval stage, checked against `spec.approvalStages[].resolvedUsers` and `spec.approvalStages[].resolvedGroups` (OIDC group claim check). Write operations (creating `ResourceRequest`, recording decisions) remain authenticated as the logged-in user via Rancher and produce per-user Kubernetes audit log entries. This split is a deliberate design choice: using a per-user watch connection would require a separate informer per active session, which does not scale. The application-layer filtering is the appropriate boundary here because the CRD objects carry no sensitive data beyond portal state — the security boundary that matters is the `create`/`patch` RBAC on the target clusters, which is enforced by the dual identity plane.

**Quota enforcement is soft at submission time.** Quota checks are performed by the Submission Handler reading the informer cache immediately before writing the `ResourceRequest`. Two near-simultaneous `create` submissions from the same user could both observe the same cache state, both pass the quota check, and both succeed — the cache has a finite resync window and no per-submission write lock exists. There is no controller-side quota re-enforcement. This is an accepted trade-off: quota is a governance guardrail, not a security control, and the race window is narrow in practice. Operators requiring hard quota enforcement should add a validating admission webhook on `ResourceRequest` that counts existing objects at write time via the API server.
